consumer-groups

Description

Files related to the "Implementing Consumer Groups" exercise.

Exercise Directory and Prerequisites

  1. This exercise requires three terminal windows with the following working directory:

    $ cd ~/training_materials/kafka/exercise-code/consumer-groups

  2. It is recommended to name the terminals producer, consA and consB.

  3. Clean and compile the code

    $ mvn package

Re-Create the newcustomers Topic with Three Partitions

The original newcustomers topic only had a single partition, so it won't serve as a good example of scaling out the consumers. We'll re-create it with three partitions. First, delete the topic:

$ kafka-topics \
    --delete \
    --zookeeper $ZK_CONNECT_STRING \
    --topic newcustomers

Next, create the topic with three partitions (so we'll have one for each broker). We'll also set a couple of properties to limit the topic's size to 1 GB so that we don't run out of disk space:

$ kafka-topics \
    --zookeeper $ZK_CONNECT_STRING \
    --create \
    --topic newcustomers \
    --partitions 3 \
    --replication-factor 3 \
    --config retention.bytes=1073742000 \
    --config segment.bytes=1073742000

Run the Producer

This producer will create a new message every two seconds, which should be slow enough to easily follow in the terminal window. In the producer terminal, run the following command:

$ mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.solution.SimpleProducer" \
    -Dexec.args="$BOOTSTRAP_SERVERS newcustomers 1000"

Run the Consumers

In the consA terminal window, start one consumer using the command below. The last value in the arguments (a) is the consumer group ID, which we're passing in from the command line instead of hardcoding as we did in earlier exercises.

$ mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.solution.SimpleConsumer" \
    -Dexec.args="$BOOTSTRAP_SERVERS newcustomers a"

In the consB terminal run a second consumer, this time using a different value (b) for the consumer group ID.

$ mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.solution.SimpleConsumer" \
    -Dexec.args="$BOOTSTRAP_SERVERS newcustomers b"

Once the consumers start running, switch back and forth between the two terminals where they're running. You should find that they both receive the same records. In other words, because they have two different values for the consumer group ID, they are operating as two unrelated applications that each receive every message on the topic.

Use Streams Messaging Manager (SMM) to View Groups' Information

Use the "Consumer Groups" section of SMM to view information about Consumer Groups A and B.

  • Which topic(s) they're reading from
  • How many consumer clients are in each group?
  • Do the clients have any lag?
  • Which host(s) the clients are running on?

Run Both Consumers in a Single Consumer Group

Kill the consumers by hitting Ctrl-C in both terminal windows. Afterwards, re-run the commands in each window, but change the consumer group ID in both cases to a.

$ mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.solution.SimpleConsumer" \
    -Dexec.args="$BOOTSTRAP_SERVERS newcustomers a"

and

$ mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.solution.SimpleConsumer" \
    -Dexec.args="$BOOTSTRAP_SERVERS newcustomers a"

You should find that each message goes to one consumer or the other; you should not see any given message going to both of them.

Observe the offsets being read by consumers

  1. Find the group.id for this consumer group
  2. Use the kafka-consumer-groups command to monitor the group's status:
kafka-consumer-groups \
--bootstrap-server $BOOTSTRAP_SERVERS \
--describe \
--group your-consumer-group-id

The output should resemble the following:

TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG CONSUMER-ID                                     HOST            CLIENT-ID
newcustomers    0          22              22              0   consumer-1-53c2780b-faa9-4748-b739-dd54d4f0f9a0 /10.0.0.106     consumer-1
newcustomers    1          21              21              0   consumer-1-53c2780b-faa9-4748-b739-dd54d4f0f9a0 /10.0.0.106     consumer-1
newcustomers    2          20              21              1   consumer-1-db1cba97-c2d8-4bcc-a930-5b6f6f31e423 /10.0.0.106     consumer-1

Note the CONSUMER-ID should be the same for at least two of the partitions. This is because there's three partitions, therefore one consumer is assigned to read partition A and B, and the other is assigned partition C.
